1.4 Expanding R’s Functionality

While base R (R as you install it, with nothing else added) is very versatile and provides a lot of functionality, most times you are going to need additional functions to simplify your data analysis and make it more reproducible.

Luckily, R provides a framework that easily allows for anyone to make their own functions, combine them into a package, and make them publicly available. Installing these packages greatly expands R’s capabilities.

In general, there are 3 main ways to install new libraries:

  1. CRAN (Comprehensive R Archive Network)- The official repository for R. All packages uploaded to CRAN have to meet certain minimum criteria and are more thoroughly tested with each new version of R.

    • To install a package from CRAN, use: install.packages("<package_name>")

  2. Bioconductor- A heavily curated, community-driven repository designed specifically for life science applications. Submissions undergo formal review by bioinformatics experts to ensure code quality. Bioconductor releases operate on a strict 6-month cycle that pairs exactly with specific R versions to prevent dependency conflicts.

    • To install a package from Bioconductor, you first need to install the package BiocManager from CRAN (install.packages("BiocManager")). Then to install packages from Bioconductor, use: BiocManager::install("<package_name>")

  3. GitHub- A code-hosting platform where people can upload packages through Git version control system. These do not have any review or update criteria so the quality of packages can vary greatly between authors.

    • To install a package from GitHub, you first need to install the package remotes from CRAN (install.packages("remotes")). Then to install packages from GitHub, use: remotes::install_github("<github_url>")

Packages are the fundamental, standardized unit of share-able code used to extend the capabilities of R. You can think of a package as a book, while R, providing the foundational functions and logic, is the entire bookshelf. Installing a package simply adds specialized book to that bookshelf

You only need to install a package once, but you need to reload it every time you start of a new R session. To continue with the bookshelf metaphor, the bookshelf is always there, along with it’s books (the installed packages). But the when you leave the room with the bookshelf, the books always have to be put back on the bookshelf. To be able to use the information contained in the books, we need to take them off the bookshelf every time we go into the room.

To do this, use the function library(<package name>), which takes the book off the bookshelf, allowing us to now use that book until we put it back on the bookshelf and leave the room ( i.e., close RStudio).

1.4.1 Installing Packages

To start, let’s install the tidyverse, a set of packages (dplyr, readr, forcats, stringr, ggplot2, tibble, lubridate, tidyr, purrr) that comprise most of the functions you’ll need for the majority of your data science tasks, all using the same common conceptual framework.

install.packages('tidyverse')

RStudio will then handle downloading and installing the packages. Once all the packages are installed, > should be visible at the bottom line of your console, indicating R is ready for your next command.

1.4.2 Loading Packages and Package Conflicts

Now that you have the tidyverse installed, let’s open it using library() as we described above. Using library() on tidyverse will load ALL of the package components at once instead of making you type each one out individually.

library(tidyverse)
## ── Attaching core tidyverse packages ───────────────────────────────────────────────────────────────────────────────────────────── tidyverse 2.0.0 ──
## ✔ dplyr     1.2.1     ✔ readr     2.2.0
## ✔ forcats   1.0.1     ✔ stringr   1.6.0
## ✔ ggplot2   4.0.3     ✔ tibble    3.3.1
## ✔ lubridate 1.9.5     ✔ tidyr     1.3.2
## ✔ purrr     1.2.2     
## ── Conflicts ─────────────────────────────────────────────────────────────────────────────────────────────────────────────── tidyverse_conflicts() ──
## ✖ dplyr::filter() masks stats::filter()
## ✖ dplyr::lag()    masks stats::lag()
## ℹ Use the conflicted package (<http://conflicted.r-lib.org/>) to force all conflicts to become errors

This specific output tells you which functions from the tidyverse were loaded, along with which functions from which package conflict with functions in base R (or from other packages you might have loaded). For example, one of the conflicts is dplyr::filter() masks stats::filter(). This means that when you use filter(), R is going to use filter() from the tidyverse package instead of filter() from the stats package (dplyr filter() masks stats filter()). Even though these two functions share the same name and serve a similar purpose, they have different arguments and syntax, so it is important to keep note of any function conflicts.

If you find yourself in the scenario where functionA() from packageX masks functionA() from packageY but you still want to use functionsA() from packageY, you can be very explicit about which package you want R to use for functionA(). In the example above, where dplyr::filter() is masking stats::filter(), say we nede to use filter() from stats instead of from dplyr. The way we accomplish this is as follows:

stats::filter()

Where you enter the exact package, followed by :: and then the exact function from that package.

1.4.3 Package Help Pages

As a part of any package, a brief description needs to be made for every function, telling the user how exactly to use that function. As described earlier, this can be accessed by typing ? followed by the function name and pressing Enter.

Every function’s help page is made up of four parts:

  • Description: a brief description of what the function does

  • Usage: an example of how you would enter the required variables

  • Arguments: a list of each of the variables (arguments) in the function along with a brief explanation of what it means and what kind of data it requires

  • Details: additional information required to fully use the package
